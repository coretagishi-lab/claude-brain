### 11:03 — 緊急: VPS メモリ逼迫対応

**状況**: 空き RAM 85MB、Swap 870MB 使用という危機的状態。

**原因**: `_kanto_cache` Python dict に関東全川データを常駐させていた（ピーク 514MB）。

**解決策**:
- `osm_rivers` / `osm_weirs` テーブルにデータを永続化（SQLite）
- bbox 空間インデックスで必要な範囲のみ SELECT
- GZip ミドルウェア追加、SW v2 でクライアントキャッシュ強化

**結果**: メモリ 254MB → 50MB（-204MB）。空き RAM 85MB → 277MB に回復。

### 16:09 — **ポリゴン方式 (natural=water) への転換**

Overpass から `natural=water` タグのポリゴンを取得し、L.geoJSON で川・池・海を塗りつぶす方式に変更。

**問題発覚**: `natural=water` は水面そのものを指すため、**橋の下はポリゴンが存在しない**（OSM の仕様上、橋は別レイヤーで管理される）。結果、橋のある箇所に「三角の隙間」が多数発生。特に隅田川・荒川などの都市河川で顕著。

### 10:40 — **橋下欠損補完の試み**

**問題の本質を整理**:
- `natural=water` ポリゴン: 水面そのものが途切れる（橋でポリゴンが切断）
- 中心線バッファリング: 幅が不自然
- GeoJSON ポリゴン: 橋の箇所で三角形の隙間が生じる

**試みた解決策**: `bridge=yes` の way から矩形ポリゴンを生成して隙間を補完する二重レイヤー構成。

**結果**: 補完ポリゴンの向きと幅の計算が複雑で、むしろ見た目が悪化。特に斜めの橋で三角形の隙間が目立つ。

---

### 11:02 — **ピクセル検出方式（採用→即廃止）**

**アイデア**: CartoDB Dark タイル画像を Canvas で再取得し、水面色（R<40 かつ B>R かつ G<60）を直接検出して塗り重ねる。GeoJSON/API 依存ゼロ。

**実装**: `createPixelDetectionHeatmap` 関数、tileLayer に `crossOrigin:'anonymous'` 追加。

**問題**:
1. CartoDB CDN が CORS ヘッダーを返さないためほぼ全タイルで `getImageData` が失敗
2. CORS キャッシュと no-CORS キャッシュが別エントリのため初回に全タイル再フェッチ発生
3. 検出できた少数タイルも水面判定が不安定（街灯反射や暗い陸地と誤検出）

**廃止判断**: 技術的に面白いが実用にならないと判断。GeoJSON ポリゴン方式に戻す。

### 16:30頃 — **タイル方式に再切り替え**

pre-generated PNG タイルを `/static/river-tiles/{z}/{x}/{y}.png` として VPS に配置し、Leaflet の `L.tileLayer` で表示する方式へ変更。

### 本日セッション — **タイル URL バグ修正（今セッション）**

**問題**: `L.tileLayer('/static/river-tiles/{z}/{x}/{y}.png', ...)` が 404。

**原因**: FastAPI の `StaticFiles` は `/` にマウントされているため、ファイルの実パスは `static/river-tiles/` だがアクセス URL は `/river-tiles/` になる。`/static/` プレフィックスが余分だった。

**確認**:
```bash
curl http://localhost:8080/static/river-tiles/13/7274/3229.png → 404
curl http://localhost:8080/river-tiles/13/7274/3229.png       → 200
```

**修正**: `index.html:774` の URL を `/river-tiles/{z}/{x}/{y}.png` に変更。VPS デプロイ・再起動済み。

## 採用した技術とその理由

| 技術 | 採用理由 |
|---|---|
| `L.tileLayer('/river-tiles/{z}/{x}/{y}.png')` | Leaflet がズームに応じて必要なタイルだけ取得。API 呼び出し不要でスムーズ |
| 国土数値情報 API（国交省） | OSM より公式・安定。APIキー不要 |
| SQLite + bbox インデックス | Python dict キャッシュ比でメモリを-200MB 削減。再起動後も保持 |
| FastAPI StaticFiles | `app.mount("/", ...)` で `static/` ディレクトリをルートに公開 |
| Service Worker バージョニング | SW v2→v7 まで毎変更時バンプ。キャッシュ起因の表示崩れを確実に防ぐ |

---

## 失敗した技術と廃止理由

| 試みた技術 | 廃止理由 |
|---|---|
| Overpass API（natural=water ポリゴン） | 橋でポリゴンが途切れる（OSM 仕様）。三角隙間多発 |
| SQLite 中心線→バッファリングポリゴン | 幅が不自然。ズームアウト時に消失 |
| CartoDB ピクセル検出 | CDN が CORS ヘッダー非対応。ほぼ全タイルで getImageData 失敗 |
| グリッド Canvas ヒートマップ | 川の「形」が出ない（点の集合になる） |
| leaflet.heat 円形グラデーション | 川の細長い形状が表現できない |
| bridge=yes 矩形ポリゴン補完 | 斜め橋で計算が複雑。見た目が悪化 |

## 2026-06-09 — ヒートマップをOSM GeoJSON方式に切り替え（ベースライン確定）

### 変更内容

PNG タイルオーバーレイ（`L.tileLayer('/river-tiles/{z}/{x}/{y}.png')`）を廃止し、OSM Overpass API から取得した実ベクターデータ（GeoJSON）を `L.geoJSON()` で直接描画する方式に切り替えた。

### 採用理由

| タイル方式（旧） | GeoJSON方式（新） |
|---|---|
| CartoDB タイル画像のピクセル色解析に依存 → CDN変更で壊れる | OSM公式データを直接描画 → 安定 |
| pre-generated PNG 324MB / 29,025ファイル | GeoJSONファイル 1ファイル 1.4MB |
| 関東域（z13〜15）のみ生成済み。全国化には数GB必要 | 1クエリで任意エリアを追加可能 |
| 橋の下が途切れる問題（タイル生成範囲外） | ポリゴン/ラインデータなので途切れなし |

### 実装詳細

**フェッチスクリプト**: `Projects/fishing-platform/fetch_osm_rivers.py`
- Overpass API クエリ（東京23区 bbox: 35.53,139.57,35.82,139.92）
- `waterway~"^(river|canal)$"` + `natural=water`（20点以上のポリゴン）
- 座標精度 4桁（約10m精度）に丸め・不要プロパティ削除

**生成ファイル**: `static/data/osm_rivers_tokyo.geojson`
- 1,418フィーチャー（川ライン 621本 + 水体ポリゴン 797件）
- 1.4MB（nginx gzip有効化で転送量 ~400KB）

**フロントエンド変更** (`static/index.html`):
- `osmRiversCache` 変数追加（ブラウザキャッシュ）
- `renderHeatmap()` を GeoJSON フェッチ → `L.geoJSON()` 描画に置き換え
- 川ライン: 青ストローク（`#0088ff`, weight:2）
- 水体ポリゴン: 青塗り（`#0055cc`, fillOpacity:0.45）
- 川名ホバーツールチップ追加（`.river-label` CSS）

### ⚠️ ベースライン宣言

> **これがヒートマップのベースライン。`osm_rivers_tokyo.geojson` は削除・変更禁止。**
> ヒートマップの表示方式（GeoJSON描画）も勝手に変更禁止。変更する場合はtagishiの明示的な指示が必要。

### タイル方式の廃止

`static/river-tiles/` フォルダ（324MB / 29,025ファイル）を削除しVPS容量を解放。
`river-tiles` エンドポイント（`/river-tiles/{z}/{x}/{y}.png`）はmain.pyに残存するが使用されない。
